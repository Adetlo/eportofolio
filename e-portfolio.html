import requests
import xml.etree.ElementTree as ET

# Real XML API from European Central Bank (ECB)
url = 'https://www.ecb.europa.eu/stats/eurofxref/eurofxref-daily.xml'
response = requests.get(url)

# Parse XML content
root = ET.fromstring(response.content)

# Navigate to exchange rates
namespace = {'gesmes': 'http://www.gesmes.org/xml/2002-08-01', 'default': 'http://www.ecb.int/vocabulary/2002-08-01/eurofxref'}
cube_root = root.find('.//default:Cube/default:Cube', namespace)

    print(f"{cube.attrib['currency']}: {cube.attrib['rate']}")

Python Web Scraping Script (BeautifulSoup4 + Requests)

Automatic search script with the keyword "Data Scientist", then parsing results to .json file:

import requests
from bs4 import BeautifulSoup
import json

url = "https://www.indeed.com/jobs?q=data+scientist&l="
response = requests.get(url)
soup = BeautifulSoup(response.content, "html.parser")

results = []
for job in soup.select(".result"):
    title = job.find("h2").text.strip() if job.find("h2") else "No title"
    company = job.find("span", class_="companyName")
    results.append({
        "title": title,
        "company": company.text.strip() if company else "Unknown"
    })

with open("data_scientist_jobs.json", "w") as f:
    json.dump(results, f, indent=4)

Automating Repetitive Cleaning

Created a Python script that can clean multiple CSV files in a folder at once:

import glob
import pandas as pd

def clean_file(path):
    df = pd.read_csv(path)
    df.columns = [c.lower().strip() for c in df.columns]
    df = df.replace([-999, 'NA'], pd.NA)
    return df.dropna()

all_files = glob.glob("data/*.csv")
cleaned = pd.concat([clean_file(f) for f in all_files])
cleaned.to_csv("consolidated_cleaned.csv", index=False)

Unit 7: Constructing Normalised Tables and Database Build

Learning Objectives

This unit focuses on transforming unstructured data tables (Un-Normalised Form / UNF) into structured relational database models using 1NF, 2NF, and 3NF normalization principles, as well as building databases based on the normalization results. Testing referential integrity and inter-table relationships is also done in practical implementation.

Key Activities & Artifacts

Table Normalization: UNF → 1NF → 2NF → 3NF

Un-Normalised Form (UNF)

Initial data had many values in one column (multivalued attributes), causing redundancy and inefficient querying.

Student Number Student Name Course Name Exam Boards Teacher Name
1001 Bob Baker Computer, Maths, Physics BCS, EdExcel, OCR Mr Jones, Ms Parker...

First Normal Form (1NF)

Separate each multivalued value into different rows:

Student Number Student Name Course Name Exam Board Teacher Name
1001 Bob Baker Computer BCS Mr Jones
1001 Bob Baker Maths EdExcel Ms Parker

Second Normal Form (2NF)

Eliminate partial dependencies by separating entities into interconnected tables:

  • Students (Student Number, Student Name, Date of Birth, Support)
  • Courses (Course ID, Course Name)
  • ExamBoards (Board ID, Board Name)
  • Teachers (Teacher ID, Teacher Name)
  • Student_Course (Student Number, Course ID, Teacher ID, Board ID, Exam Score)

Third Normal Form (3NF)

Remove transitive dependencies – ensure all non-key attributes depend only on primary key:

  • All tables are separated with foreign keys and no columns depend on non-key attributes
MySQL Database Implementation

I implemented the normalized database into MySQL with 5 main tables and 4 relationships:

CREATE TABLE Students (
  StudentNumber INT PRIMARY KEY,
  StudentName VARCHAR(100),
  DateOfBirth DATE,
  Support BOOLEAN
);

CREATE TABLE Courses (
  CourseID INT AUTO_INCREMENT PRIMARY KEY,
  CourseName VARCHAR(50)
);

CREATE TABLE ExamBoards (
  BoardID INT AUTO_INCREMENT PRIMARY KEY,
  BoardName VARCHAR(50)
);

CREATE TABLE Teachers (
  TeacherID INT AUTO_INCREMENT PRIMARY KEY,
  TeacherName VARCHAR(100)
);

CREATE TABLE Student_Course (
  ID INT AUTO_INCREMENT PRIMARY KEY,
  StudentNumber INT,
  CourseID INT,
  BoardID INT,
  TeacherID INT,
  ExamScore INT,
  FOREIGN KEY (StudentNumber) REFERENCES Students(StudentNumber),
  FOREIGN KEY (CourseID) REFERENCES Courses(CourseID),
  FOREIGN KEY (BoardID) REFERENCES ExamBoards(BoardID),
  FOREIGN KEY (TeacherID) REFERENCES Teachers(TeacherID)
);
Testing and Integrity Validation

Performed several queries to ensure:

  • Table relationships are valid (foreign key constraints work well)
  • Data retrieval is efficient using JOIN
  • No duplication or data anomalies after migration from UNF

Example query to retrieve Bob Baker's scores:

SELECT s.StudentName, c.CourseName, sc.ExamScore
FROM Students s
JOIN Student_Course sc ON s.StudentNumber = sc.StudentNumber
JOIN Courses c ON sc.CourseID = c.CourseID
WHERE s.StudentName = 'Bob Baker';

I learned that normalization is not just a technical practice, but an important strategy to ensure scalability, consistency, and efficiency in data management.

The process of separating entities and identifying functional dependencies helped me see data relationships more conceptually. My biggest challenge was structuring multivalue data from text form into relational—especially separating Course, Exam Board, and Teacher which initially seemed unified.

Database implementation in MySQL strengthened my understanding of referential integrity and how logical design can be physically realized in real database systems.

Unit 11: Transaction Management and Disaster Recovery

Learning Objectives

Understand how database systems maintain consistency and integrity in the face of system failures or crashes. Topics include ACID transactions, concurrency control, logging, checkpointing, backup strategies (GFS), and business continuity planning in real-time environments.

Key Activities & Artifacts

Atomic Fund Transfer Simulation (MySQL)
START TRANSACTION;
-- Deduct from Alice
UPDATE accounts SET balance = balance - 100 WHERE name = 'Alice';
-- Add to Bob
UPDATE accounts SET balance = balance + 100 WHERE name = 'Bob';
COMMIT;

If a failure occurs between the two queries or before the COMMIT, the transaction will be rolled back automatically, preserving database integrity.

This simulation demonstrates atomicity and consistency, two core ACID properties essential for any critical system such as banking or inventory management.

Grandfather-Father-Son (GFS) Backup Strategy
  • Son: Daily backups (rotated weekly)
  • Father: Weekly backups (retained for a month)
  • Grandfather: Monthly snapshots for long-term archival

This hierarchical strategy optimizes storage and ensures multi-level rollback options—key in regulated sectors such as banking, healthcare, and telecoms.
